## 變更摘要

### 做了什麼

**1. 診斷證明照片可刪除 / 再新增（原本上傳一張後就回不去）**
- 根因：翻譯員上傳照片後 slot 變 `completed`，而 `MySchedules` 的 `showActions` 條件把按鈕在 completed 時隱藏 → 無法再開 modal。
- 後端：新增照片清單（含 id）與刪除端點，翻譯員版做擁有權檢查、管理員版為 surrogate；刪除時 best-effort 移除磁碟檔，且當該 slot 照片歸零時狀態退回 `pending`。
- 前端：`DiagnosisUploadModal` 改寫為「管理照片」modal（檢視既有照片、逐張刪除、依剩餘額度新增、操作後不自動關閉）；completed slot 改顯示「管理照片」按鈕。

**2. 新增翻譯員 email 重複時顯示錯誤訊息**
- 後端原本已正確回傳 409 `EMAIL_TAKEN`（三語系皆有翻譯），問題純在前端 `handleCreate` 把錯誤吞掉顯示通用「失敗」。
- 改用 `extractApiError(err)` 顯示後端具體原因。

**3. 照片改永久保存**
- `PHOTO_RETENTION_DAYS` 預設由 90 改為 0；`CleanupService` 在 retention ≤ 0 時直接跳過、永不刪除。
- 背景清理 cron 仍保留，僅在明確設定正整數天數時才會清。

### 注意事項
- **新增 API 端點**（皆走既有 JWT 中介層）：
  - 翻譯員：`GET /api/checkins/diagnosis/photos?schedulePatientId=`、`DELETE /api/checkins/diagnosis/photos/:photoId`
  - 管理員：`GET /api/admin/diagnosis/photos?schedulePatientId=`、`DELETE /api/admin/diagnosis/photos/:photoId`
  - list 端點刻意用 query param，避免與 `:photoId` 造成 gin wildcard 衝突。
- **行為變更**：刪除某 slot 的最後一張照片會把狀態退回 `pending`（可重新補傳或標記未到）。
- **永久保存**：硬碟空間會持續增長，請確保容量並安排備份；若日後要恢復自動清理，把 `PHOTO_RETENTION_DAYS` 設為正整數即可。
- DB schema 無變更（沿用既有 `diagnosis_photos` 表）。
