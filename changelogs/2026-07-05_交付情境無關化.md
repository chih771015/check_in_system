## 變更摘要

### 做了什麼
釐清了「給 image 還是給原始碼」的疑問，並把交付文件從「診所專用單一劇本」
重寫成**情境無關**，讓不知道環境的客戶能自己決定怎麼架：

- **概念釐清（對談紀錄）**：
  - Docker image = 把程式 + 其執行環境凍結成可攜、開機即跑的唯讀模板；container 是其執行實例。
  - image 的參數**可以**用環境變數設定，所以「不給 image」**不是**因為參數不能設。
  - 真正原因：本系統是多服務組合（仍需 compose 編排）、image 交付要 registry/大 tar、
    綁 CPU 架構、且客戶無法自行修改。對「未知環境的客戶」，image 更綁死而非更方便。
- **docs/PRODUCTION_DEPLOY.md 重寫**：
  - 新增系統架構圖，點出「反向代理是必要的」（前端固定用相對路徑 `/api`）。
  - 三大決定：A 怎麼跑（Docker Compose / 裸機）、B 資料庫（內建容器 / 雲端·自備）、
    C 怎麼對外（自架反代 / 雲端 LB / Cloudflare Tunnel / 純內網——**完全由客戶決定**）。
  - 補完整「不用 Docker 裸機路線」逐步教學：Go binary build、前端 `npm run build`、
    自架反向代理（/api、/uploads、SPA fallback、放大上傳上限）、同源免 CORS 提醒。
  - 選用整合（SMTP / Google Sheets / LINE / Jaeger）全部標為 opt-in。
- **HANDOVER.md**：中英雙語新增「你要自己決定的三件事」；對外公開定位為客戶自決、
  tunnel 僅為選項之一；指向裸機路線。

### 影響範圍
- 修改檔案：`docs/PRODUCTION_DEPLOY.md`（整篇重寫）、`HANDOVER.md`。
- 純文件，未動任何程式邏輯或架構（架構本就支援這些選項，只是文件先前寫死在單一情境）。

### 注意事項
- 對外公開、網域、HTTPS 憑證一律由客戶端負責，系統不強制任一方案。
- 接雲端 / 自備 DB 時，走 Docker 需在 `docker-compose.prod.yml` 註解掉內建 postgres 與其 depends_on。
