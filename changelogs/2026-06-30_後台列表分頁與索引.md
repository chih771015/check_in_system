## 變更摘要

### 做了什麼

把後台四個會「全撈整表、再前端切頁」的列表全部改成 server-side 真分頁，並補上
篩選/排序用得到的索引。起因：打卡紀錄頁原本後端無任何上限，整張 checkins 表全撈，
資料只增不減，越用越慢。盤查後發現排班（篩選模式無上限）、翻譯員、管理員也是同樣
的「假分頁」寫法，一併改掉。

四頁一致的做法：
- 後端 repo 查詢回傳 `(rows, total)`，`PageSize > 0` 才套 `Limit/Offset`，
  `PageSize <= 0` 維持「不分頁取全部」供匯出 / 夜間提醒 / 下拉選單使用。
- handler 解析 `page` / `pageSize`，一律回傳 `{ data, total, page, pageSize }` 信封。
- 前端 API 讀信封，頁面改用 AntD Table 的 server-side pagination
  （`current/pageSize/total/onChange`），篩選變更時 `setPage(1)`。

額外修正：
- 打卡日期篩選原本用 `DATE(checkin_time)` 包欄位，導致 PostgreSQL 用不到索引；
  改為對原欄位的半開區間 `>= from AND < to+1day`。
- `checkins.checkin_time`、`schedules.created_at` 補上索引。
- `getAdmins` 先前 `r.data.data` 對「單鍵被自動 unwrap 的回應」取值會是 undefined，
  改回傳分頁信封後一併修正。

### 注意事項

- **回應格式變更（破壞性）**：四個列表 API 由「裸陣列」改為 `{ data, total, page, pageSize }`
  信封。前端對應呼叫已全部更新；若有外部呼叫者需同步調整。
- 翻譯員下拉選單仍走 `getTranslators`（不帶分頁＝取全部），與管理表格的
  `getTranslatorsPaged` 分開，避免下拉只拿到一頁。
- 匯出（Excel / Google Sheet）與夜間提醒任務不帶 PageSize，維持取全部，不受分頁影響。
- 索引由 GORM AutoMigrate 於啟動時建立；資料量小，建立瞬間完成。
- 驗證：後端 `go test ./...` 全過；前端 `tsc -b` 通過、`npm test` 102 案全過。
