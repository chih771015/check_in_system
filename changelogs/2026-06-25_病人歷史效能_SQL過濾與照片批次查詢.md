## 做了什麼

針對 `PatientService.GetHistory` / `buildHistoryEntries`：

### #2 消除照片 N+1
- 新增 `DiagnosisPhotoRepository.FindBySchedulePatientIDs(ids)`：以 `IN (...)` 一次查回多個 slot 的照片（依 uploaded_at ASC），空 ids 直接回空。
- `buildHistoryEntries` 改為：收集所有 sp_id → 單次批次查照片 → 在 Go 端用 map 依 sp_id 分組。原本是每筆看診各跑一次照片查詢（N+1）。

### #5 日期過濾下推 SQL
- `buildHistoryEntries` 新增 `from, to` 參數，於 SQL 加 `schedules.date` 條件，只撈保留的列；不再全量撈回再於 Go 端過濾。
- `GetHistory` 移除 in-Go 過濾迴圈，改為對（已由 SQL 過濾的）entries 加總 actual。

### #6 統一日期區間慣例
- 歷史頁是「使用者挑選的閉區間」[from, to]；為了在 SQL 安全表達（sqlite 的 date 以 RFC3339 字串儲存，直接 `<= to` 會漏掉當天 `T00:00:00Z` 的列），新增 `nextDay()` 把閉區間上界轉成半開排除界：`date >= from AND date < (to+1天)`。
- 如此三條「依日期區間加總」路徑（歷史、年度 hint、當月 banner）現在一律走 SQL `>=/<` 半開比較，不再有「in-Go 閉區間 vs SQL 半開」兩套機制。年度/當月維持整段半開 [from, to)，歷史以 nextDay 轉換維持對使用者的閉區間語意。
- `to` 非法（無法解析）時跳過上界條件（best-effort，與原本字典序比較對非法輸入同樣寬鬆，無行為退化）。
