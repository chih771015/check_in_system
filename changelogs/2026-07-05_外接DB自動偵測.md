## 變更摘要

### 做了什麼
讓 `deploy-prod.sh` **依 `DB_HOST` 自動判斷**要不要啟動內建 PostgreSQL 容器，
接雲端／自備 DB 時不必再手動編輯 compose：

- **拆檔（overlay 架構）**：
  - `docker-compose.prod.yml`（base）移除內建 postgres 服務、backend 對 postgres 的
    `depends_on`、硬寫的 `DB_HOST=postgres`、以及 `postgres_data` volume。
  - 新增 `docker-compose.localdb.yml`（overlay），只有用內建 DB 時才疊加，加回上述內容。
- **關鍵修正**：base 原本在 backend `environment` 硬寫 `DB_HOST: postgres`，會**蓋掉**
  `.env` 的值 → 外接 DB 永遠連到內建容器。移到 overlay 後，外接模式的 `DB_HOST` 由 `.env` 決定。
- **腳本自動判斷**：`DB_HOST=postgres`（或空）→ 內建模式（疊加 overlay、自動產生
  `DB_PASSWORD`）；其他值 → 外接模式（不疊加、不亂產密碼、驗證必填）。驗證規則、
  結尾印出的維運指令都依模式帶正確的 `-f` 檔。

### 影響範圍
- 修改：`docker/docker-compose.prod.yml`、`scripts/deploy-prod.sh`、`docs/PRODUCTION_DEPLOY.md`。
- 新增：`docker/docker-compose.localdb.yml`。
- 開發用 `docker-compose.yml` 與 expose overlay 不受影響（postgres 仍內嵌，dev 用）。

### 驗證方式
- `bash -n scripts/deploy-prod.sh` 通過。
- `docker compose config` 實測兩模式：
  - 內建：服務含 postgres，backend `depends_on` 同時有 jaeger(started)+postgres(healthy)，`DB_HOST=postgres`。
  - 外接（`DB_HOST=my-rds.example.com`）：服務無 postgres，backend 只依賴 jaeger，`DB_HOST` 為 `.env` 的值、未被覆蓋。

### 注意事項
- **內建 DB** 模式下手動下 compose 指令要**兩個檔一起帶**（base + localdb），否則 `down`
  會把 postgres 留成孤兒；最保險是抄腳本結尾印出的指令。漏帶時用
  `docker compose -f docker/docker-compose.prod.yml down --remove-orphans` 補清。
- 外接模式 `DB_PASSWORD` 是那顆 DB 的真實密碼，腳本不會自動產生，留空會被擋。
- `backend/.env` 仍被 git 追蹤一事（前報告提及）尚未處理。
