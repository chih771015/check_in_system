## 變更摘要

### 問題描述

在網頁上按「刪除排班」，若該排班已有打卡紀錄，操作失敗並在 Jaeger 顯示：

```
ERROR: update or delete on table "schedules" violates foreign key constraint
"fk_checkins_schedule" on table "checkins" (SQLSTATE 23503)
```

### 根因

`checkins.schedule_id` 有 Foreign Key 指向 `schedules.id`。直接執行 `DELETE FROM schedules WHERE id=?` 時，PostgreSQL 偵測到 `checkins` 仍有引用該排班的紀錄，依 FK 規則拒絕刪除。

### 修復方式（Application-level cascade）

刪除排班前，在 service 層先刪除該排班的所有 checkin 紀錄，再刪排班本身。優先選此方案而非修改 DB schema（`ON DELETE CASCADE`），避免 migration 風險。

### 做了什麼

- `repository/checkin_repo.go`：新增
  - `DeleteByScheduleID(scheduleID uint) error`
  - `DeleteByScheduleIDs(scheduleIDs []uint) error`（供群組刪除使用）
- `repository/schedule_repo.go`：新增
  - `IDsByRecurrenceGroup(groupID string) ([]uint, error)`（取群組內所有排班 ID）
- `service/schedule_service.go`：
  - `Delete()`：刪排班前先呼叫 `DeleteByScheduleID`
  - `DeleteRecurrenceGroup()`：先取所有排班 ID → 批次刪 checkin → 再刪排班

### 影響範圍

| 範圍 | 檔案 |
|------|------|
| 後端 Repository | `backend/internal/repository/checkin_repo.go`、`schedule_repo.go` |
| 後端 Service | `backend/internal/service/schedule_service.go` |

### 注意事項

- 刪除排班屬於**不可逆操作**，打卡紀錄一併刪除後無法復原
- 若未來需要保留歷史打卡，可改為軟刪除（加 `deleted_at` 欄位）或先封存再刪
