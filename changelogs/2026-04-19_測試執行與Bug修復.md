### 發現的 Bug

#### Bug 1 — `recurrenceGroupId` 序列化問題（已修復）

**問題**：非重複性排班的 `recurrenceGroupId` 在 JSON response 中完全省略，應為 `null`。  
**根因**：`backend/internal/dto/schedule.go` 的 `ScheduleResponse` 中 `RecurrenceGroupID *string` 標記了 `omitempty`，nil pointer 被 JSON marshaler 略去。  
**修復**：移除 `omitempty`，nil pointer 現在正確序列化為 `null`。

#### Bug 2 — Audit Log 分頁行為（已確認為設計行為，無需修復）

**描述**：`?page=0` 與 `?page=1` 回傳相同結果。  
**分析**：後端實作 1-indexed 分頁（page=1 為第一頁），page<=0 正規化為 1 是防禦性設計。前端也從 `page=1` 開始。page=0 與 page=1 回傳相同結果屬於預期行為，測試使用 page=1 與 page=2 驗證分頁正常運作，結果正確。

#### Bug 3 — Export Schedule `enabled:false` 無法持久化（已修復）

**問題**：透過 API 將 `enabled` 設為 `false`，回傳成功，但 GET 查詢仍顯示 `enabled: true`。  
**根因**：GORM 的 `Create` 預設跳過 zero value（`bool` 的 zero value 是 `false`）。INSERT 語句中 `enabled` 欄位被省略，DB column default `true` 填入，ON CONFLICT 的 `EXCLUDED.enabled` 也因此是 `true`。  
**修復**：在 `Upsert` 呼叫前加入 `Select("admin_id", "frequency", ..., "enabled", "updated_at")`，強制 GORM 將所有指定欄位（包括 zero value）寫入 INSERT 語句。
