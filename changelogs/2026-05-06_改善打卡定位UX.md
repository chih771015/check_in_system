## 變更摘要

### 問題描述

翻譯員進入打卡頁面時，定位授權彈窗不夠顯眼，使用者容易誤按「拒絕」或完全沒看到。拒絕後畫面只有小字錯誤，沒有引導如何解決，使用者卡住不知道怎麼辦。

### 根因分析

1. 原本頁面一載入就靜默呼叫 `getCurrentPosition`，瀏覽器的系統對話框在行動裝置上很容易被忽略
2. 錯誤訊息只有 `<Typography.Text type="danger">` 小字，不夠醒目
3. `PERMISSION_DENIED` / `POSITION_UNAVAILABLE` / `TIMEOUT` 三種不同錯誤用同一則文字，無法給出具體操作指引

### 修復

#### 新增 `useGeolocation` hook（`hooks/useGeolocation.ts`）

定義 6 種狀態機：`idle → requesting → success / denied / unavailable / timeout`

- mount 時先查詢 `navigator.permissions.query({ name: 'geolocation' })`
  - `granted` → 直接自動取得，不打擾使用者
  - `denied` → 直接進入 `denied` 狀態，不再彈瀏覽器對話框
  - `prompt` → 保持 `idle`，等使用者主動點按鈕（permission priming）

#### 新增 `GeoStatusBlock` 共用元件（`CheckIn.tsx` 匯出）

依狀態顯示不同的 Ant Design `Alert`：

| 狀態 | 顯示 |
|------|------|
| `idle` | 黃色 Alert + 說明文字 + **「授權定位」**按鈕 |
| `requesting` | 藍色載入動畫 |
| `success` | 綠色打勾 + 座標 |
| `denied` | 紅色 Alert + iOS / Android / Chrome 三平台開啟步驟 |
| `timeout` | 橘色 Alert + 重試按鈕 |
| `unavailable` | 橘色 Alert + 重試按鈕 |

- 「確認送出」按鈕在狀態非 `success` 時 `disabled`，防止沒定位就送出

#### `MakeupCheckIn.tsx` 同步更新

引用相同的 `useGeolocation` hook 與 `GeoStatusBlock`，行為完全一致。

### 注意事項

- `navigator.permissions` 在部分舊版 iOS Safari（< 16）不支援，此時 hook 會保持 `idle` 狀態，讓使用者手動點授權按鈕，行為退化安全
- 定位功能需要 **HTTPS**，透過 cloudflared tunnel 對外時為 HTTPS，正常運作
- 若使用者已拒絕定位，重新整理頁面後仍會顯示 `denied` 狀態，直到使用者在瀏覽器設定中手動重新允許
