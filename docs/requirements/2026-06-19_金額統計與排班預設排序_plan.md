## 共用後端基礎：金額統計查詢

多項需求都要對 `schedule_patients.actual_amount` 做加總，集中在一個 repository helper，避免重複 SQL。

**新增 `SchedulePatientRepository` 方法（皆含 `WithCtx`）：**

```go
// 當月（或任意月）所有病人實付總額：依 schedules.date 落在 [from, to)
func (r *SchedulePatientRepository) SumActualByDateRange(from, to time.Time) (int64, error)

// 單一病人全時段實付總額
func (r *SchedulePatientRepository) SumActualByPatient(patientID uint) (int64, error)

// 單一病人在日期區間的實付總額（歷史頁區間查詢 / 年度查詢共用）
func (r *SchedulePatientRepository) SumActualByPatientDateRange(patientID uint, from, to time.Time) (int64, error)

// 多病人一次取全時段實付總額（列表用，避免 N+1）：map[patientID]sum
func (r *SchedulePatientRepository) SumActualByPatients(patientIDs []uint) (map[uint]int64, error)
```

所有查詢 `JOIN schedules ON schedules.id = sp.schedule_id`，依 `schedules.date` 過濾。`no_show` 因 `actual_amount=0` 自然不計。

> SQLite(memory) 與 Postgres 的 date 比較：用 `time.Time` 參數綁定，GORM 會處理；測試以固定日期建構資料避免時區問題。

## 風險 / 注意

- **SQLite vs Postgres date 比較**：repo 測試用 `time.Time` 綁定 + 固定日期，避免字串格式差異（既有 `buildHistoryEntries` 已處理 `T` 截斷，可參考）。
- **N+1**：列表總額務必用 `SumActualByPatients(ids)` 批次查，而非逐列查。
- **Excel 匯入相容**：第 5 欄只在匯出端加，import header 常數與解析維持 4 欄。
- **路由 wildcard**：新增 `/patients/:id/actual-total` 與既有 `/patients/:id/history` 同屬 GET `:id` 子路徑，Gin 可共存。
- **橫幅效能**：當月總額查詢為單一 aggregate，成本低；每次進後台頁載入一次即可。
